# Artifact for "Language Mechanization Frameworks into Real-World Programming Language Specifications in the Wild"

## Installing from Source on Ubuntu 22.04

Clone the artifact repository and setup submodules.

```bash
$ git clone https://github.com/oopsla-p4spectec/artifact
$ git submodule update --init --recursive
```

### Installing HOL4P4

Navigate to the HOL4P4 project directory, and run the install script to install HOL4P4 and its dependencies.

```bash
$ cd hol4p4
$ ./scripts/install.sh
```

### Installing Petr4

Navigate to the project directory and install the following dependencies:

```bash
$ cd petr4
$ sudo apt update && \
  sudo apt install -y --no-install-recommends \
    pkg-config \
    m4 \
    libgmp-dev \
    opam \
    ca-certificates \
    curl \
    unzip
```

Initialize opam if it has not been yet:

```bash
$ opam init --disable-sandboxing -y
```

Create a local switch at project root, and install opam depdencies.

```bash
$ opam switch create . --empty
$ eval $(opam env)

# petr4-013.export is a list of opam packages with their versions pinned.
$ opam switch import petr4-013.export -y
$ eval $(opam env)
```

Build and install petr4.

```bash
$ dune build
$ dune install
```

### Installing P4-SpecTec

Navigate to the P4-Spectec project directory, and install the following dependencies:

```bash
$ cd p4-spectec
$ sudo apt update && \
  sudo apt install -y --no-install-recommends \
    pkg-config \
    libgmp-dev \
    opam
```

Initialize opam if it has not been yet:

```bash
$ opam init --disable-sandboxing -y
```

Create an opam switch for ocaml 5.1.0, and install opam dependencies:

```bash
$ opam switch create 5.1.0
$ eval $(opam env --switch=5.1.0)

$ opam install dune bignum 'menhir=20240715' 'menhirLib=20240715' core core_unix bisect_ppx yojson ppx_deriving_yojson -y
$ eval $(opam env)
```

Build the project, which will produce `p4-spectec` and `p4-spectec-test` under the project root directory.

```bash
$ make release spec-test
```

## Getting Started with Docker

First, clone the repository and setup submodules.

```bash
$ git clone https://github.com/oopsla-p4spectec/artifact
$ git submodule update --init --recursive
```

Build an image from the `Dockerfile` provided by this repository.

```bash
$ docker build -t p4-eval-harness:latest .
```

Run the docker container, and execute `bash` inside it:

```bash
# launch a docker container in detached state.
$ docker run -dit --name artifact p4-eval-harness

# run a shell inside the container
$ docker exec -it artifact bash
```

## P4-SpecTec

P4-SpecTec is a submodule of this repository. Within the submodule, the `spec-concrete` directory contains the mechanized P4 specification.

## Reproducing RQ1, RQ2, and RQ3

### RQ1: Adequacy of Algorithmic Inference Rules

#### Executable backend tests

We run tests on positive and negative type checker tests for the type checker backend:

* `Pos`: 1,307 positive type checker tests from the `p4c` repository.
* `Neg`: 564 negative type checker tests from the `p4c` repository.

For the interpreter backend, we run tests on five test suites spanning three architectures: V1Model, eBPF, and PSA. The test suites are labeled as follows:

* `V1Model`: 203 P4-STF pairs for the V1Model architecture from the `p4c` repository.
* `eBPF`: 17 P4-STF pairs for the eBPF architecture from the `p4c` repository.
* `V1Model+`: 1,982 additional P4–STF pairs for the V1Model architecture, generated by feeding P4 programs from `p4c` into P4Testgen, configured to produce up to 10 STF files per program.
* `eBPF+`: 144 additional P4–STF pairs for the eBPF architecture, generated by feeding P4 programs from `p4c` into P4Testgen, configured to produce up to 10 STF files per program.
* `PSA`: 26 P4-STF pairs for the PSA architecture from the `p4c` repository.

The excluded tests are listed in `testdata/excludes`, and the patched tests are listed in `testdata/patches`.

Execute the following command to run the P4-SpecTec P4 type checker and interpreter backend tests.

```bash
# if installed from Docker, just use rq1 instead of scripts/rq1
$ scripts/rq1 -i p4include -e testdata/excludes -p testdata/patches -t testdata -l logs --spectec p4-spectec
```

#### Document backend

The generated HTML and PDF specifications are in `docs/P4-16-spec-release.html` and `docs/P4-16-spec-release.pdf` in the P4-SpecTec submodule.

### RQ3: Comparison with Existing Formalizations

In this section, we compare P4-SpecTec with Petr4 and HOL4P4.
Each project can be found in the root directory of the docker container.

```
/
├── HOL4P4
├── petr4
└── p4-spectec
```

We provide `rq3`, a script to simplify running test suites across projects. The following is an overview of its options.

```bash
# if installed from Docker, just use rq3 instead of scripts/rq3

# Run static semantics tests or compare test results.
$ scripts/rq3 -i p4include -e testdata/excludes/ -t testdata -l logs \
   --spectec [P4-SpecTec Root] --petr4 [Petr4 Root] --hol4p4 [HOL4P4 Root] \
   static {p4spectec,petr4,compare} {pos,neg}

# Run dynamic semantics tests or compare test results.
$ scripts/rq3 -i p4include -e testdata/excludes/ -t testdata -l logs \
   --spectec [P4-SpecTec Root] --petr4 [Petr4 Root] --hol4p4 [HOL4P4 Root] \
   dynamic {p4spectec,petr4,hol4p4,compare} {v1model,ebpf,v1model+,ebpf+}
```

#### Static Semantics

We compare the implementation of P4 static semantics from P4-SpecTec and Petr4 by running both positive and negative type checker tests on the two projects. HOL4P4 is excluded from this comparison, as it is unable to execute its P4 type system mechanization.

Execute the following command to compare test results from P4-SpecTec and Petr4.
```bash
# if installed from Docker, just use rq3 instead of scripts/rq3
# Run positive test suite and compare results across projects
$ scripts/rq3 -i p4include -e testdata/excludes/ -t testdata -l logs \
   --spectec [P4-SpecTec Root] --petr4 [Petr4 Root] --hol4p4 [HOL4P4 Root] \
   static compare pos

Generating CSV...
Comparison result written to [...]/positive-static-results.csv.

# if installed from Docker, just use rq3 instead of scripts/rq3
# Run negative test suite and compare results across projects
$ scripts/rq3 -i p4include -e testdata/excludes/ -t testdata -l logs \
   --spectec [P4-SpecTec Root] --petr4 [Petr4 Root] --hol4p4 [HOL4P4 Root] \
   static compare neg

Generating CSV...
Comparison result written to [...]/negative-static-results.csv.
```

Running a test suite with a single project is also possible. For example, the following command runs the negative typecheck test suite with Petr4, and produces test results in a json file under `/petr4`.

```bash
# if installed from Docker, just use rq3 instead of scripts/rq3
$ scripts/rq3 -i p4include -e testdata/excludes/ -t testdata -l logs \
   --spectec [P4-SpecTec Root] --petr4 [Petr4 Root] --hol4p4 [HOL4P4 Root] \
   static petr4 neg
```

#### Dynamic Semantics

We compare the implementation of P4 dynamic semantics from P4-SpecTec, Petr4 and HOL4P4 by running the test suites `V1Model`, `eBPF`, `V1Model+`, and `eBPF+`.

Execute the following command to compare test results from P4-SpecTec, Petr4 and HOL4P4.
```bash
# if installed from Docker, just use rq3 instead of scripts/rq3
# Run positive test suite and compare results across projects
$ scripts/rq3 -i p4include -e testdata/excludes/ -t testdata -l logs \
   --spectec [P4-SpecTec Root] --petr4 [Petr4 Root] --hol4p4 [HOL4P4 Root] \
   dynamic compare <v1model|ebpf|v1model+|ebpf+>

Generating CSV...
Comparison result written to [...]/<suite>-dynamic-results.csv.
```

> [!WARNING]
> The `v1model+` test suite has over 1900 tests. Running it on HOL4P4 will take a long time to finish, even up to 12+ hours.

To run a specific test suite with a single project, pass the project name to the second argument of the `rq3`. For example, execute the following command to run the `ebpf` test suite with HOL4P4.
```bash
# if installed from Docker, just use rq3 instead of scripts/rq3
$ scripts/rq3 -i p4include -e testdata/excludes/ -t testdata -l logs \
   --spectec [P4-SpecTec Root] --petr4 [Petr4 Root] --hol4p4 [HOL4P4 Root] \
   dynamic hol4p4 ebpf
```
